大型电商网站的商品详情页系统的核心架构思路 商品服务 MQ 缓存数据生产服务 ehcache redis nginx(html模板 ---> 本地缓存 ---> 前端) 店铺服务 MySQL 品牌服务 1. 数据修改(价格修改)入库(MySQL)时向MQ发送修改信息 2. 缓存数据生产服务 ---> 监听 ---> MQ (如果页面的数据有变更 及时监听 并写入缓存中 提供高并发 高性能的访问) 3. 缓存数据生产服务 ---> 将监听到的数据写入到redis中 4. 本地缓存有效时间10分钟 ---> 过时就去redis中读取 5. 如果html模板变了 不用全量重新渲染　直接将最新的html模板推送到nginx服务器 6. 大量的情况下　数据是直接从nginx本地取缓存 没有网络请求的开销　没有业务逻辑　渲染到模板中 html页面返回 缓存架构 ---> redis架构 高并发，高可用，海量数据，备份，随时可以恢复，缓存架构如果要支撑这些要点，首先呢，redis就得支撑 redis架构，每秒钟几十万的访问量QPS，99.99%的高可用性，TB级的海量的数据，备份和恢复，缓存架构就成功了一半了 最最简单的模式，无非就是存取redis，存数据，取数据 支撑你的缓存架构，最基础的就是redis架构 解决各种各样高并发场景下的缓存面临的难题，缓存架构中不断的引入各种解决方案和技术，解决高并发的问题 解决各种各样缓存架构本身面临的高可用的问题，缓存架构中引入各种解决方案和技术，解决高可用的问题
